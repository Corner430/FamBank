# Feature Specification: FamBank 多家庭多租户产品化

**Feature Branch**: `002-multi-tenant-platform`
**Created**: 2026-03-01
**Status**: Draft
**Input**: FamBank 从单家庭工具演进为 To C 多家庭平台：手机号+短信验证码认证、家庭创建与邀请、多子女独立账户、租户数据隔离、家长聚合 dashboard。对应章程 V4.0（第1.1条、第9-11条）和宪法 2.0.0（原则 VIII 租户隔离、原则 IX 认证与身份）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 手机号注册与登录 (Priority: P1)

新用户通过手机号和短信验证码注册平台账号。已注册用户通过手机号和短信验证码登录。系统不使用密码或 PIN 码，验证码是唯一的身份认证方式。开发阶段使用假验证码（固定码如 123456）以绕过短信网关。

**Why this priority**: 认证是所有功能的前提。没有用户体系，后续的家庭创建、成员管理、数据隔离都无法实现。这是多租户架构的地基。

**Independent Test**: 可以通过输入手机号获取验证码、输入验证码完成注册/登录、验证返回有效会话凭证来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户未注册, **When** 输入手机号 13800138000 并请求验证码, **Then** 系统发送（开发阶段模拟发送）6位数字验证码，有效期5分钟
2. **Given** 验证码已发送, **When** 用户输入正确验证码, **Then** 系统创建新账号并返回登录会话凭证，用户进入"无家庭"状态
3. **Given** 验证码已发送, **When** 用户输入错误验证码, **Then** 系统拒绝并提示"验证码错误"，允许重试
4. **Given** 验证码已发送, **When** 超过5分钟后用户输入验证码, **Then** 系统拒绝并提示"验证码已过期，请重新获取"
5. **Given** 用户已注册, **When** 输入已注册手机号并请求验证码后正确输入, **Then** 系统直接登录（而非重复注册），返回会话凭证
6. **Given** 同一手机号, **When** 60秒内重复请求验证码, **Then** 系统拒绝并提示"请60秒后再试"
7. **Given** 同一手机号, **When** 连续5次输入错误验证码, **Then** 系统锁定该手机号验证15分钟，提示"错误次数过多，请稍后再试"

---

### User Story 2 - 创建家庭 (Priority: P1)

已注册用户可以创建一个新家庭。创建者自动成为该家庭的家长（parent 角色）。创建家庭时需要填写家庭名称。创建完成后系统初始化该家庭的默认配置参数（分流比例、利率等，使用章程默认值）。

**Why this priority**: 家庭是整个业务的容器，与认证并列 P1。没有家庭，就没有孩子账户、结算、交易等一切业务功能。

**Independent Test**: 可以通过已登录用户创建家庭、验证家庭创建成功、验证创建者角色为 parent、验证默认配置参数已初始化来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户已登录且无家庭, **When** 用户填写家庭名称"张家"并提交创建, **Then** 系统创建家庭，用户自动成为该家庭的 parent 角色，会话凭证更新为包含家庭信息
2. **Given** 家庭刚创建, **When** 查询家庭配置, **Then** 返回章程默认参数（分流比例 15/30/55、B利率 2.0/1.2/0.3、C利率 5%、罚倍数 2、赎回费率 10% 等）
3. **Given** 用户已属于一个家庭, **When** 用户尝试创建新家庭, **Then** 系统拒绝并提示"您已属于一个家庭，不可重复创建"
4. **Given** 用户已登录, **When** 用户提交空白家庭名称, **Then** 系统拒绝并提示"家庭名称不能为空"

---

### User Story 3 - 邀请成员加入家庭 (Priority: P1)

家长可以生成邀请码，分享给其他用户以加入家庭。被邀请人使用邀请码加入时，由家长指定其角色（parent 或 child）。孩子加入家庭后，系统自动为该孩子创建独立的 A/B/C 三账户（初始余额为 0）。

**Why this priority**: 邀请机制是多成员家庭的入口。与 US1/US2 并列 P1，因为单人家庭无意义——至少需要一个家长和一个孩子才能使用系统。

**Independent Test**: 可以通过家长生成邀请码、新用户使用邀请码加入、验证角色分配和账户初始化来独立测试。

**Acceptance Scenarios**:

1. **Given** 家长已登录, **When** 家长生成邀请码并指定角色为 child、名称为"小明", **Then** 系统返回一个唯一邀请码（含有效期，默认7天）
2. **Given** 邀请码已生成（角色 child、名称"小明"）, **When** 新注册用户输入邀请码加入, **Then** 用户成为该家庭的 child 角色、显示名称为"小明"，系统自动为其创建 A/B/C 三账户（余额均为 0）
3. **Given** 邀请码已生成（角色 parent）, **When** 新注册用户输入邀请码加入, **Then** 用户成为该家庭的 parent 角色，不创建 A/B/C 账户（家长无个人账户）
4. **Given** 邀请码已过期（超过7天）, **When** 用户尝试使用该邀请码, **Then** 系统拒绝并提示"邀请码已过期"
5. **Given** 邀请码已被使用, **When** 另一用户尝试使用同一邀请码, **Then** 系统拒绝并提示"邀请码已失效"（一码一用）
6. **Given** 用户已属于某个家庭, **When** 尝试使用邀请码加入另一个家庭, **Then** 系统拒绝并提示"您已属于一个家庭"

---

### User Story 4 - 多子女独立账户与入账 (Priority: P1)

一个家庭可以有多个孩子，每个孩子独立持有一套 A/B/C 三账户。家长为某个孩子录入收入时，系统将收入按比例分流至**该孩子自己的**三个账户。不同孩子的账户完全隔离，互不影响。

**Why this priority**: 多子女独立核算是本次产品化的核心变更之一。与前三个 US 一起构成了多租户最小可用集合——有了认证、家庭、邀请和独立账户，系统就能开始正常的业务操作。

**Independent Test**: 可以通过为家庭添加两个孩子，分别录入不同金额的收入，验证各自 A/B/C 账户余额独立变化、互不干扰来测试。

**Acceptance Scenarios**:

1. **Given** 家庭有两个孩子（小明、小红），各自 A/B/C 均为 0, **When** 家长为小明录入收入 100 元, **Then** 小明的 A=15 元、B=30 元、C=55 元；小红的 A/B/C 均不变
2. **Given** 小明 A=15 元, **When** 家长为小红录入收入 200 元, **Then** 小红的 A=30 元、B=60 元、C=110 元；小明的 A 仍为 15 元
3. **Given** 家长已登录, **When** 家长在收入录入界面选择目标孩子, **Then** 系统显示家庭内所有孩子列表供选择
4. **Given** 家庭只有一个孩子, **When** 家长录入收入, **Then** 系统自动选定唯一的孩子，无需手动选择

---

### User Story 5 - 家长聚合 Dashboard (Priority: P2)

家长登录后看到的首页是聚合 Dashboard，展示家庭内所有孩子的账户概览信息。家长可以从 Dashboard 切换到查看某个孩子的详细账户页面（与原有单孩子视图一致）。

**Why this priority**: Dashboard 是多子女场景下家长的核心操作入口。但它依赖 US1-4 的基础设施已建好，且不影响资金操作的核心逻辑。

**Independent Test**: 可以通过创建一个有多个孩子（且有不同余额）的家庭，以家长身份登录后验证 Dashboard 显示所有孩子摘要信息、点击可进入详细页面。

**Acceptance Scenarios**:

1. **Given** 家庭有两个孩子（小明、小红），小明 A=100/B=200/C=300，小红 A=50/B=150/C=250, **When** 家长查看 Dashboard, **Then** 页面展示每个孩子的名字和三账户余额摘要
2. **Given** Dashboard 显示中, **When** 家长点击小明的卡片/行, **Then** 进入小明的详细账户页面，显示其 A/B/C 完整信息（余额、利息池、交易记录等）
3. **Given** 家庭有3个孩子, **When** 家长查看 Dashboard, **Then** 页面同时展示3个孩子的摘要信息，包含家庭总资产汇总
4. **Given** 家庭暂无孩子成员, **When** 家长查看 Dashboard, **Then** 显示引导提示"还没有孩子加入，请生成邀请码邀请成员"

---

### User Story 6 - 孩子视图（仅看自己） (Priority: P2)

孩子登录后只能看到自己的 A/B/C 三账户信息和交易记录，看不到其他兄弟姐妹的任何数据，也看不到聚合 Dashboard。

**Why this priority**: 数据隔离的前端体现。孩子不应看到家庭其他成员的财务数据，这是隐私和公平性的要求。

**Independent Test**: 可以通过以孩子身份登录，验证只能看到自己的账户和交易，无法访问其他孩子的数据或家长 Dashboard。

**Acceptance Scenarios**:

1. **Given** 小明以 child 角色登录, **When** 查看首页, **Then** 显示小明自己的 A/B/C 三账户余额和最近交易，与原有单孩子视图基本一致
2. **Given** 小明以 child 角色登录, **When** 尝试通过任何方式（URL 篡改、API 调用等）访问小红的账户数据, **Then** 系统拒绝并返回无权限错误
3. **Given** 小明以 child 角色登录, **When** 查看交易记录, **Then** 仅显示小明自己的交易记录

---

### User Story 7 - 多子女独立结算 (Priority: P2)

月度结算时，系统对家庭内每个孩子独立执行结算 SOP（C派息→B溢出→B计息→违约划转）。每个孩子的结算基于其各自的账户余额、愿望清单 P_active、违约记录等，互不影响。

**Why this priority**: 结算是 001-fambank-core 的核心功能，多子女场景下需要确保逐个独立结算。依赖 US4（多子女账户）。

**Independent Test**: 可以通过为两个孩子设定不同的账户状态，执行月度结算后验证各自的结算结果独立正确（如一个孩子有溢出而另一个没有）。

**Acceptance Scenarios**:

1. **Given** 小明 C=1200 元、小红 C=600 元, **When** 家长触发月度结算, **Then** 小明 C 派息 5.00 元入小明 A、小红 C 派息 2.50 元入小红 A，互不影响
2. **Given** 小明 B=2000 且 P_active=1500、小红 B=800 且 P_active=1500, **When** 月度结算, **Then** 小明 B 溢出 200 元至 C、小红 B 无溢出
3. **Given** 小明有违约记录（50 元入 A）、小红无违约, **When** 月度结算, **Then** 仅小明执行违约划转，小红不受影响
4. **Given** 家庭有3个孩子, **When** 结算过程中第2个孩子结算失败, **Then** 该孩子的结算回滚，但不影响其他已成功或未开始的孩子结算

---

### User Story 8 - 租户数据隔离 (Priority: P1)

不同家庭的数据严格隔离。任何用户只能访问自己所属家庭的数据。系统在服务层面强制过滤，确保即使前端或 API 参数被篡改，也不可能跨家庭访问数据。

**Why this priority**: 与 US1 并列 P1。数据隔离是 To C 产品的底线要求（章程第11条、宪法原则 VIII），不满足则不能上线。

**Independent Test**: 可以通过创建两个独立家庭，各有成员和账户数据，然后以家庭 A 的成员身份尝试访问家庭 B 的数据（通过 API 传入非本家庭的 ID），验证全部被拒绝。

**Acceptance Scenarios**:

1. **Given** 家庭A（张家）有孩子小明、家庭B（李家）有孩子小红, **When** 张家家长查询账户列表, **Then** 只返回小明的账户，不包含小红
2. **Given** 张家家长的会话凭证, **When** 通过 API 传入小红的 user_id 查询账户, **Then** 系统拒绝，返回"无权限"（而非"未找到"，以防信息泄露）
3. **Given** 张家家长的会话凭证, **When** 通过 API 传入李家的 family_id 查询配置, **Then** 系统拒绝
4. **Given** 两个家庭各自进行了多笔交易, **When** 张家成员查询交易记录, **Then** 仅返回张家的交易，绝不包含李家的记录

---

### User Story 9 - 现有业务功能适配多租户 (Priority: P2)

001-fambank-core 中已实现的所有业务功能（愿望清单、购买、消费、违约、赎回、配置、公告等）需要适配为多租户模式。所有操作都必须在家庭和孩子维度下执行，不影响原有业务逻辑的正确性。

**Why this priority**: 这是将原有功能从单租户迁移到多租户的"翻新"工作。依赖 US1-4 和 US8 的基础设施。

**Independent Test**: 可以通过在多租户环境下执行原有的全部业务流程（录入收入、创建愿望清单、购买消费、违约处理、赎回、结算、配置变更），验证功能与单租户版本行为一致。

**Acceptance Scenarios**:

1. **Given** 小明有愿望清单, **When** 小明发起购买, **Then** 从小明的 B 账户扣款（逻辑与 001 一致），小红不受影响
2. **Given** 张家配置分流比例为 20/30/50, **When** 张家为小明录入收入, **Then** 按 20/30/50 分流；李家仍按默认 15/30/55
3. **Given** 张家家长公告利率变更, **When** 变更生效, **Then** 仅影响张家，不影响其他家庭
4. **Given** 小明有违约记录, **When** 查询违约列表, **Then** 只返回小明的违约，不含小红或其他家庭成员

---

### Edge Cases

- 用户注册后既没有创建家庭也没有加入家庭，处于"无家庭"状态——系统强制显示 onboarding 页面，仅提供"创建家庭"和"输入邀请码"两个入口，屏蔽所有其他导航和业务操作
- 家庭只有家长没有孩子——Dashboard 显示引导提示，结算时无需执行（无孩子即无账户）
- 家长邀请码发出后、被使用前，家长将邀请码作废——邀请码应支持撤销
- 孩子加入家庭后立即触发结算——新孩子的 A/B/C 均为 0，结算结果为空操作
- 手机号格式校验：仅接受中国大陆手机号格式（11位，1开头）
- 验证码并发：同一手机号在一个验证码有效期内再次请求——返回"请等待当前验证码过期后重试"或直接复用
- 家庭创建后家长退出——目前不支持退出家庭（简化 MVP），后续迭代考虑
- 多家长同时为同一个孩子录入收入——正常处理，各自生成独立的交易记录，通过并发控制保证余额一致
- 结算过程中某个孩子的结算失败——仅回滚该孩子的结算，不影响其他孩子
- JWT token 中的 family_id 与请求路径中的 family_id 不一致——拒绝请求

## Requirements *(mandatory)*

### Functional Requirements

**认证**

- **FR-001**: 系统 MUST 支持通过手机号 + 短信验证码进行注册和登录，不使用密码或 PIN 码
- **FR-002**: 系统 MUST 在开发阶段提供假验证码模式（固定验证码 123456），通过配置切换
- **FR-003**: 验证码 MUST 为6位数字，有效期5分钟，同一手机号60秒内不可重复请求
- **FR-004**: 系统 MUST 在同一手机号连续5次验证失败后锁定15分钟
- **FR-005**: 系统 MUST 自动识别手机号是否已注册，已注册则走登录流程，未注册则走注册流程
- **FR-006**: 系统 MUST 返回包含 user_id、family_id（可为空）和 role 的会话凭证，access token 有效期24小时，refresh token 有效期30天
- **FR-006a**: 无家庭用户登录后 MUST 强制进入 onboarding 页面，仅提供"创建家庭"和"输入邀请码"两个入口，屏蔽所有业务导航
- **FR-006b**: 系统 MUST 支持通过 refresh token 静默续期 access token，refresh token 过期后用户需重新验证码登录

**家庭管理**

- **FR-007**: 已登录用户 MUST 可以创建家庭，提供家庭名称，创建者自动成为 parent 角色
- **FR-008**: 系统 MUST 在家庭创建时自动初始化章程默认配置参数（分流比例、利率、罚倍数等）
- **FR-009**: 每个用户 MUST 最多属于一个家庭
- **FR-010**: 家长 MUST 可以生成邀请码，指定被邀请人角色（parent 或 child）和显示名称，邀请码有效期7天、一码一用
- **FR-011**: 用户 MUST 可以通过邀请码加入家庭，系统按邀请码指定的角色分配
- **FR-012**: 孩子加入家庭时，系统 MUST 自动为其创建独立的 A/B/C 三账户（初始余额 0）
- **FR-013**: 家长 MUST 可以撤销未使用的邀请码

**多子女独立账户**

- **FR-014**: 每个孩子 MUST 独立持有 A/B/C 三账户，不同孩子的账户在数据层面严格隔离
- **FR-015**: 收入录入 MUST 指定目标孩子，分流至该孩子自己的 A/B/C 账户
- **FR-016**: 月度结算 MUST 对家庭内每个孩子独立执行完整 SOP，互不影响
- **FR-017**: 某个孩子的结算失败 MUST 仅回滚该孩子的结算，不影响其他孩子

**数据隔离**

- **FR-018**: 系统 MUST 在服务层对所有数据查询强制注入 family_id 过滤，不依赖调用方传递
- **FR-019**: 系统 MUST 拒绝任何跨家庭的数据访问请求，返回无权限错误
- **FR-020**: 会话凭证 MUST 携带 family_id，所有 API 请求 MUST 基于凭证中的 family_id 过滤数据
- **FR-021**: 同一家庭内，孩子 MUST 只能查看自己的账户和交易数据
- **FR-022**: 同一家庭内，家长 MUST 可以查看所有孩子的账户和交易数据

**家长 Dashboard**

- **FR-023**: 家长登录后 MUST 看到聚合 Dashboard，展示所有孩子的账户余额摘要和家庭总资产
- **FR-024**: 家长 MUST 可以从 Dashboard 切换到查看某个孩子的详细账户页面
- **FR-025**: 孩子登录后 MUST 直接进入自己的账户页面，不显示 Dashboard 或其他孩子信息

**业务适配**

- **FR-026**: 001-fambank-core 中已实现的所有业务功能（愿望清单、购买、消费、违约、赎回、配置、公告）MUST 适配为在家庭和孩子维度下执行
- **FR-027**: 配置参数（分流比例、利率等）MUST 按家庭隔离，各家庭独立管理
- **FR-028**: 所有交易记录 MUST 包含 family_id 和 user_id（操作对象孩子），支持按家庭和孩子维度查询

**去除 PIN 码**

- **FR-029**: 系统 MUST 移除现有的 PIN 码认证机制，统一使用手机号 + 验证码认证
- **FR-030**: 原有需要 PIN 码确认的操作（如结算、赎回审批）MUST 改为基于角色权限控制（家长角色即可执行）

### Key Entities

- **用户 (User)**: 平台注册用户，通过手机号标识，可属于一个家庭并持有 parent 或 child 角色。
- **家庭 (Family)**: 多租户的核心实体，包含家庭名称、创建时间、邀请码。所有业务数据归属于某个家庭。
- **邀请码 (Invitation)**: 家长生成的一次性邀请凭证，包含目标角色、被邀请人显示名称、有效期、使用状态。
- **短信验证码 (SmsCode)**: 发送到手机号的验证码，包含手机号、验证码、过期时间、使用状态。
- **账户 (Account)**: 归属于某个孩子的 A/B/C 账户，新增孩子所属关系。一个孩子恰好有 3 个账户。
- **配置 (Config)**: 归属于某个家庭的参数配置，每个家庭独立管理，从章程默认值初始化。
- **交易记录 (Transaction)**: 新增家庭和孩子归属信息，其他字段与 001 一致。

## Clarifications

### Session 2026-03-01

- Q: 孩子的显示名称由谁在何时设定？ → A: 家长在生成邀请码时为孩子预设名字
- Q: 无家庭状态的用户看到什么？ → A: 强制 onboarding 页面，仅显示"创建家庭"和"输入邀请码"两个选项，屏蔽所有其他导航
- Q: JWT token 过期时间？ → A: Access token 24小时过期 + Refresh token 30天

## Assumptions

- 每个用户最多属于一个家庭（不支持一人多家庭，简化 MVP）
- 家长没有个人的 A/B/C 账户，家长是管理者角色
- 目前不支持退出或解散家庭（后续迭代考虑）
- 目前不支持移除家庭成员（后续迭代考虑）
- 开发阶段短信验证码使用假验证码模式（固定码 123456），生产环境切换到腾讯云 SMS
- 手机号仅支持中国大陆格式（11位，1开头）
- 邀请码为一次性使用，过期后自动失效
- 结算按孩子粒度独立执行，某个孩子结算失败不影响其他孩子
- 现有单家庭数据需要迁移方案（不在本 spec 范围内，需单独处理）
- 去除 PIN 码后，所有权限控制完全依赖角色（parent/child）+ JWT 会话

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户可以在2分钟内完成注册、创建家庭、邀请第一个孩子的完整流程
- **SC-002**: 家庭内多个孩子的账户余额在任何操作后保持完全独立——对孩子 A 的操作不会改变孩子 B 的任何余额
- **SC-003**: 跨家庭数据访问的成功率为 0%——所有跨家庭请求均被拒绝
- **SC-004**: 家长可在 Dashboard 上一屏查看所有孩子的账户摘要，切换到某个孩子的详细页面在1次点击内完成
- **SC-005**: 所有原有业务功能（入账、结算、购买、违约、赎回、配置）在多租户环境下的行为与单租户版本完全一致（通过原有测试用例回归验证）
- **SC-006**: 验证码发送和验证的成功率在正常使用场景下达到 99%（不含故意输错的场景）
- **SC-007**: 月度结算支持对家庭内最多10个孩子独立执行，单个孩子的结算失败不影响其他孩子
